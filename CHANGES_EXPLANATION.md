# Progressive Dynamic Filter Pushdown for Sort-Merge Join

This document provides a detailed technical explanation of the changes implemented to support progressive min/max dynamic filter pushdown for `SortMergeJoinExec` in DataFusion.

## 1. Core Architectural Change: The "Upward Handshake"
The most significant part of this change wasn't just adding filters to the Join, but fixing the **physical optimizer's propagation chain**. DataFusion's physical optimizer works in two passes:
1.  **Downward Pass (`gather_filters_for_pushdown`)**: Determines which filters can reach the scan.
2.  **Upward Pass (`handle_child_pushdown_result`)**: If a child (like a Scan) accepts a filter, it returns an `updated_node`. Every parent above it **must** then update itself to point to this new child, or the link is discarded.

### Operators Fixed:
The following built-in DataFusion operators were updated to correctly propagate the `updated_node` during the upward pass:
- `SortExec`
- `SortPreservingMergeExec`
- `ProjectionExec`
- `RepartitionExec`
- `FilterExec` (for the `Post` phase)
- `CoalesceBatchesExec`
- `GlobalLimitExec` & `LocalLimitExec`
- `CooperativeExec`

Without these fixes, dynamic filters generated by a Join would be "blocked" by any of these operators sitting between the Join and the Scan (or above the Join).

## 2. Sort-Merge Join Implementation (`SortMergeJoinExec`)

### Progressive Pruning Logic
Unlike Hash Join, which must wait for the entire build side to finish, Sort-Merge Join (SMJ) leverages the **sorted property** of its inputs. As each partition advances, the "head" of the stream provides a progressively tighter lower bound (for Ascending) or upper bound (for Descending).

### Shared Accumulator (`SharedSortMergeBoundsAccumulator`)
A new coordination module was added to implement **"consensus of the slowest progress"**:
- **Consensus**: In a parallel join, a global filter can only advance as far as the **slowest** partition. The accumulator tracks the current head of every partition and calculates the `min` (for Ascending) or `max` (for Descending) across all active partitions.
- **Sudden Death**: If one side of the join is completely exhausted (all partitions finished or the input was empty), the accumulator updates the opposite side's filter to `Literal(false)`. This signals the data source to terminate its scan immediately.

### Optimization: Batch-Boundary Updates
Reporting the "head" for every single row is computationally expensive due to mutex locking. The implementation was optimized to report the current join key only when a **new batch** is polled from the input streams. This provides the majority of the pruning benefit with minimal overhead.

## 3. Metrics and Visibility

### New Metrics
- `dynamic_filter_updates`: A counter that tracks how many times the min/max bound was tightened during execution. This provides visibility into the filter's activity even if the final state is `false`.

### EXPLAIN Support
The `DisplayAs` implementation for `SortMergeJoinExec` was updated. Standard `EXPLAIN` and `EXPLAIN ANALYZE` outputs will now show:
- `left_dynamic_filter=...`
- `right_dynamic_filter=...`

*Note: In `EXPLAIN ANALYZE`, it is expected to see these filters as `false` at the end of the run, as the join has reached the end of its input and triggered "Sudden Death" pruning.*

## 4. Verification and Testing

### Unit Tests
Five new unit tests were added to `datafusion/physical-plan/src/joins/sort_merge_join/tests.rs`:
1.  `test_sort_merge_join_dynamic_filter_initial_bound`: Verifies the first bound publication.
2.  `test_sort_merge_join_dynamic_filter_progressive_tightening`: Verifies that bounds tighten at batch boundaries.
3.  `test_sort_merge_join_dynamic_filter_consensus`: Verifies that multiple parallel partitions coordinate correctly.
4.  `test_sort_merge_join_dynamic_filter_sudden_death_empty`: Verifies immediate termination when a side is empty.
5.  `test_sort_merge_join_dynamic_filter_descending`: Verifies that the logic correctly flips for Descending sort orders.

### Integration Tests
Existing tests in `core_integration` were updated:
- `test_smj_filter_pushdown_supported`: Now verifies that `Left`, `Right`, `Semi`, and `Anti` joins correctly propagate filters.
- `test_smj_filter_pushdown_unsupported`: Confirms that `Full` joins still correctly block pushdown (as they require all rows from both sides).
